# 后端错误码与通用能力归一规格

> 文档类型：SPEC
> 状态：已完成
> 更新时间：2026-02-27
> 适用范围：skuare-svc/internal, skuare-svc/README.md
> 关联计划文件路径：plan/2026-02-27-01/backend-util-errcode-unification.md

## 背景与目标
- 当前后端错误码散落在 `internal/http/error.go` 字符串字面量中，不利于统一维护。
- 部分业务无关的通用逻辑（例如通用拒绝错误、参数错误识别）与业务模块耦合，复用成本较高。
- 目标是将错误码统一到 `util/errcode.go`，并将其它业务无关、可复用的能力下沉到 `util` 目录。

## 需求范围
- 在后端新增 `skuare-svc/internal/util/errcode.go`，统一维护 HTTP 错误码常量。
- 将业务无关的通用错误与判定逻辑迁移到 `skuare-svc/internal/util`。
- 调整 `internal/http`、`internal/authz` 对应引用，保证行为不变。
- 更新后端模块文档，明确错误码统一定义位置与对外语义。

## 非目标
- 不改动 API 路径、请求/响应结构与状态码策略。
- 不引入新的错误码种类或鉴权策略。
- 不改造 store 层业务语义错误（如技能版本不存在）的判定来源。

## 用户验收标准与风险
- 验收标准：
  - 错误码定义集中在 `util/errcode.go`。
  - 通用无业务语义能力从业务模块下沉到 `util`。
  - 后端全量单元测试通过，且 `go build ./...` 通过。
  - 用户确认错误码归一与 util 目录整理符合预期。
- 风险：
  - 错误变量迁移后若仍存在旧引用，可能导致编译失败或错误映射失效。
  - 参数错误识别逻辑位置变更可能造成行为偏差。

## 需求优化建议
- 建议 1：后续补充 `mapError` 的表驱动单测，覆盖所有错误码映射分支。
  - 收益：降低回归风险，增强重构稳定性。
  - 代价：新增测试维护成本。
  - 风险：若错误分类策略调整频繁，测试用例需同步维护。
- 建议 2：中期将错误码与 OpenAPI 文档建立自动一致性校验。
  - 收益：减少文档与实现漂移。
  - 代价：需要新增校验脚本与 CI 配置。
  - 风险：初期规则设计不完善可能带来误报。
