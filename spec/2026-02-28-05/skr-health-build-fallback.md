# skr health 构建回退修复规格

> 文档类型：SPEC
> 状态：进行中
> 更新时间：2026-02-28
> 适用范围：project-wide
> 关联计划文件路径：plan/2026-02-28-05/skr-health-build-fallback.md

## 背景与目标
- 当前根目录 `skr` 包装脚本会在检测到 CLI 源码新于 `dist/.build-stamp` 时强制执行 `npm run build`。
- 在未安装 `typescript` 依赖、但仓库里已存在可执行 `skuare-cli/dist/index.js` 的环境下，`skr health` 会因为 `tsc: not found` 提前失败，无法进入真实的健康检查逻辑。
- 目标是在保留源码变更后优先重建的前提下，为已有预构建产物场景增加安全回退，确保 `skr health` 等命令不会因缺少本地 TS 构建工具而误报失败。

## 需求范围
- 调整根目录 `skr` 的构建逻辑：当自动重建失败且已有可执行 `dist/index.js` 时，允许输出明确告警并继续使用现有产物运行。
- 当 `dist/index.js` 不存在时，仍保持失败退出，避免静默掩盖真正不可运行的状态。
- 同步更新相关 README/计划文档，说明 `skr` 的预构建回退行为。

## 非目标
- 不修改 CLI 命令语义或后端健康检查接口。
- 不在本需求中引入新的包管理策略或自动安装 TypeScript 依赖。
- 不把所有构建失败都无条件吞掉；无预构建产物时仍应明确报错。

## 用户验收标准与风险
- 用户验收标准：
  - 在缺少 `tsc` 的环境中执行 `./skr health` 不再因为 `tsc: not found` 失败。
  - 当服务未启动时，命令进入真实网络请求阶段，并返回当前已有的网络错误。
  - CLI/SVC 全量检查命令通过。
- 风险：
  - 若源码已修改但预构建产物过旧，回退运行可能与最新源码不一致。
  - 构建失败原因若不是工具缺失而是代码错误，需要避免被静默吞掉。

## 需求优化建议
- 建议后续在发布流程中显式产出与校验 CLI 预构建版本信息，将 `dist` 与源码版本关联起来，降低“源码已更新但仍回退旧产物”的可见性风险。
