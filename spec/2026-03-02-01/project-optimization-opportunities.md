# 项目整体优化机会规格

> 文档类型：SPEC
> 状态：已完成
> 更新时间：2026-03-02
> 适用范围：project-wide
> 关联计划文件路径：plan/2026-03-02-01/project-optimization-opportunities.md

## 背景与目标
- 当前仓库已经形成 `skuare-cli`、`skuare-svc`、根目录脚本与多份文档并行演进的格局，功能主链路可用，但存在代码组织、帮助文本、测试覆盖与扩展边界不够稳定的问题。
- 本需求目标不是立即改造全部问题，而是基于现状形成一份可执行、可追踪、可继续拆分的项目级优化规格，为后续迭代提供统一输入。
- 本规格希望覆盖以下维度：
  - 代码风格与一致性。
  - CLI 与 SVC 功能体验。
  - 模块边界与扩展性。
  - 测试、构建与验证闭环。
  - 文档、脚本与工程协作流程。

## 需求范围
- 对当前仓库进行项目级梳理，并输出一份结构化优化清单，至少覆盖以下方面：
  - 代码风格与可维护性：
    - 降低超大文件、重复解析逻辑、重复终端输出逻辑带来的维护成本。
    - 明确命令层、配置层、传输层、领域层的职责边界。
    - 收敛重复帮助文案、重复路径解析、重复错误格式化入口。
  - 功能能力与用户体验：
    - 识别 CLI 命令语义不够稳定、参数演进后文档和脚本容易漂移的问题。
    - 识别下载、安装、缓存、覆盖写入、冲突提示等用户关键链路中的可优化点。
    - 识别初始化配置、多工具目录管理、正则查询和安装反馈中的易混淆点。
  - 扩展性与架构演进：
    - 识别 CLI 现有命令实现中对远端拉取、依赖解析、文件写入、交互式输入的耦合点。
    - 识别 SVC 现有 service/store 边界中过薄封装、全量扫描与索引更新职责不清的问题。
    - 识别未来引入缓存策略、更多存储后端、批量操作和更强查询能力时的阻碍因素。
  - 测试与质量保障：
    - 识别 CLI 自动化测试覆盖过窄、缺少端到端回归、缺少文档与帮助一致性校验的问题。
    - 识别构建与验证流程中容易遗漏的模块和命令链路。
  - 文档与工程流程：
    - 识别根 README、模块 README、CLI help、Makefile 帮助信息之间的漂移风险。
    - 识别 Spec/Plan 与实际实现之间缺少回溯链接、缺少长期优化路线图的问题。
- 优化清单需要包含优先级视角，至少区分：
  - 应优先处理的高收益问题。
  - 可分阶段推进的结构性优化。
  - 中长期能力建设项。

## 非目标
- 不在本需求中直接重构 CLI/SVC 代码实现。
- 不在本需求中直接新增或修改服务端 API。
- 不在本需求中直接调整现有命令行为、存储格式或线上兼容策略。
- 不要求本轮把优化项全部拆成独立实施 spec/plan；本轮只沉淀项目级规格输入。

## 用户验收标准与风险
- 验收标准：
  - `spec/2026-03-02-01/project-optimization-opportunities.md` 明确覆盖代码风格、功能、扩展性、测试、文档与流程等维度。
  - 优化点不是泛泛罗列，而是能对应到仓库现状中的具体问题类型。
  - 文档可作为后续拆分子需求的上游规格输入，便于继续创建更小的专项 spec/plan。
  - 本轮文档补充不破坏仓库既有构建、测试与说明体系。
- 风险：
  - 若优化点只停留在抽象口号，后续无法拆分为可执行任务。
  - 若不区分优先级，容易把局部文案问题与结构性技术债混为一谈。
  - 若缺少与现有代码现状的映射，后续实施时仍可能重复分析。

## 需求优化建议
- 建议 1：为项目级优化清单增加优先级和影响面标记。
  - 收益：便于后续按“高收益低风险”先落地。
  - 代价：需要额外维护分类和排序。
  - 潜在风险：若分类标准不稳定，后续可能频繁调整。
- 建议 2：将项目级优化清单按“命令层、配置层、存储层、测试层、文档层”拆成后续专题 spec。
  - 收益：便于并行推进，降低单次改动范围。
  - 代价：需要额外维护多份 spec/plan。
  - 潜在风险：拆分过细会增加管理成本。
- 建议 3：后续为帮助文案、README、Makefile 帮助和 CLI 命令注册增加一致性校验。
  - 收益：可以提前发现命令参数和文档漂移。
  - 代价：需要引入额外校验脚本或测试。
  - 潜在风险：如果校验设计过严，会增加文档演进阻力。

## 当前观察到的优化机会
### P0：应优先处理
- 命令实现过度集中：`skuare-cli/src/commands/query.ts`、`skuare-cli/src/commands/write.ts` 体量明显偏大，内部混合参数解析、HTTP 调用、文件系统写入、依赖解析、输出组装与交互逻辑，后续继续加命令会放大维护成本。
- 帮助与脚本文案漂移：根 `README.md`、`skuare-cli/README.md`、`Makefile` 帮助、CLI `printHelp()` 中已有命令语义不完全同步，参数演进后容易出现过时说明。
- CLI 测试覆盖不足：当前 CLI 自动化测试基本只覆盖 `build`，`get/peek/list/init/publish/format/delete` 等关键路径缺少回归，文档改动或重构后回归风险高。
- 安装链路缺少更稳健的写入语义：`get` 当前直接落盘目标目录，缺少更明确的 cache/install 边界、原子写入、回滚或冲突策略扩展点。

### P1：适合分阶段推进
- 配置与路径解析重复：CLI 中存在多处地址规范化、路径推导、颜色输出和选项解析逻辑，职责分散在 `index.ts`、`query.ts`、`init.ts`、`resolver.ts`、`http/client.ts`。
- 命令注册与帮助文案双维护：命令列表和 usage 示例存在多个维护入口，新增命令或改参数时容易漏改。
- Service 层过薄：`skuare-svc/internal/service/skill_service.go` 当前几乎只是直通 store，后续如果增加鉴权上下文、审计、缓存或批量能力，缺少清晰的领域编排层。
- Store 查询路径偏粗：`GetSkill`、`GetVersion` 通过 `List("")` 和全量遍历索引取数据，在技能规模变大后查询成本和职责边界都不够理想。
- 缺少更清晰的领域模型：CLI 输出对象、远端数据结构、文件内容解析结果之间存在重复转换，未来扩展 JSON 输出和脚本消费能力时不够稳定。

### P2：中长期建设项
- 缺少 CLI-SVC 一体化集成测试和示例回放。
- 缺少针对 README、help、Makefile 帮助的一致性检查。
- 缺少项目级性能、并发、兼容性回归基线。
- 缺少围绕缓存、批量安装、依赖冲突策略、更多存储后端的专项路线图。
