# `skr init` 不再暴露服务端存储目录规格

> 文档类型：SPEC
> 状态：已完成
> 更新时间：2026-03-01
> 适用范围：skuare-cli, docs
> 关联计划文件路径：plan/2026-03-01-06/init-remote-storage-server-owned.md

## 背景与目标
- 当前 `skr init` 会把 `remote.storageDir` 作为客户端配置项暴露并写入配置文件。
- 但 `remote.storageDir` 实际上是服务端启动参数（如 `--spec-dir`）决定的服务端内部存储路径，不应由 CLI 初始化流程声明、维护或推断。
- 目标是收敛 CLI 配置语义：客户端只保存连接信息与签名凭证，不再把服务端存储目录当作客户端可配置字段。

## 需求范围
- 调整 `skr init` 交互、快照输出与保存逻辑，移除 `remote.storageDir` 的展示、编辑与写入。
- 调整 CLI 类型与配置默认值，避免新配置默认携带 `remote.storageDir`。
- 去掉 `get` 命令中基于 `remote.storageDir` 的客户端共享目录推断，避免客户端对服务端目录做假设。
- 更新相关 README/技术文档，明确服务端存储目录仅由服务端启动参数决定。
- 保持对历史配置文件的读取兼容，不因为旧配置里残留 `remote.storageDir` 而报错。

## 非目标
- 不修改服务端 `--spec-dir` / `SKUARE_SPEC_DIR` 的行为。
- 不在本需求中新增服务端能力发现接口或目录查询接口。
- 不调整 `get` 命令的安装目录规则，仍保持本地局部仓库布局不变。

## 用户验收标准与风险
- 验收标准：
  - `skr init` 不再提示或打印 `remote.storageDir`。
  - 新写出的 CLI 配置文件不包含 `remote.storageDir`。
  - 旧配置中即使包含 `remote.storageDir`，CLI 仍能正常解析并执行常规命令。
  - CLI 构建检查与服务端历史单测全量通过。
- 风险：
  - 失去客户端共享目录推断后，`get` 命令不再输出旧的 `shared_local_dir=true` 结果。
  - 文档若未同步更新，用户仍可能误以为 CLI 可以决定服务端仓库目录。

## 需求优化建议
- 建议后续由服务端提供只读运行参数摘要接口，仅暴露安全字段，例如地址、模式与仓库根摘要，用于 CLI 展示诊断信息。
- 收益：避免客户端本地猜测服务端配置。
- 代价：需要新增服务端接口与兼容性维护。
- 风险：若暴露字段设计不当，可能泄漏不必要的部署信息。
